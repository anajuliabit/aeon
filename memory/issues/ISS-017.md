root_cause: |
  Two `run:` blocks bind `inputs.chain` via direct GitHub Actions template
  interpolation instead of the `env:` indirection pattern used in
  `messages.yml` after the 2026-04-11 incident. Lines:
    - chain-runner.yml:41  → `CHAIN="${{ inputs.chain }}"`
    - chain-runner.yml:416 → `CHAIN="${{ inputs.chain }}"`
  Both sit inside `run: |` shell bodies, so if a malicious chain name
  containing shell metacharacters were ever passed, it would expand into
  the executable shell context rather than into a quoted env-var.
  Real-world exposure today is low: the only trigger paths are
  `workflow_dispatch` (requires repo write auth) and concurrency-group
  expansion (line 17, a YAML-level field — not executed). But the
  anti-pattern is the same class that produced the 2026-04-11 messages.yml
  incident, and the messages.yml fix (env: indirection, lines 587-591
  binding `_INPUT_MESSAGE`/`_EVENT_ACTION`/etc. then reading `$_INPUT_MESSAGE`
  in the shell) is the canonical remediation.
fix: |
  Mirror the messages.yml fix. Replace direct interpolation with env:
  binding. For both `run:` blocks (lines 36-409 and 411-468):
    env:
      _CHAIN: ${{ inputs.chain }}
    run: |
      CHAIN="$_CHAIN"
  Concurrency group at line 17 (`group: aeon-chain-${{ inputs.chain }}`)
  is a YAML expression evaluated by the runner, not shell — leave as-is.
  Same for `run-name:` at line 2.
