## What happened

The 2026-05-23 afternoon `reppo-swarm` run queued three NO vote intents
for datanet 9: pods 46 (Sairen / OpenAI Gym), 74 (clickbait AI trading
video), and 150 (TradingGymAI Pipeline framework). `postprocess-reppo.sh`
ran dry-runs in `dry_run_only=true` mode; all three reverted identically:

```
output: {"error":{"code":"POD_NOT_VALID_FOR_EPOCH",
                  "message":"Pod <id> is not valid for the current voting epoch.",
                  "hint":"Check `reppo query pod <id>` to verify validity;
                         pod publishers may need to republish."}}
```

This is a distinct failure mode from ISS-003 (the `code: UNKNOWN`
wrapping) — the nested CLI error code is well-defined and tells us
exactly what went wrong: these pods are present in the cache but the
current voting epoch will not accept votes against them.

## Impact

- 0 votes cast on-chain for this run.
- This is recurrent: prior runs also queued votes against pods that
  later failed dry-run (e.g. pod 46 also failed on the morning run).
  Until the agent filters by epoch validity, vote intents will continue
  to be wasted.

## Fix

Two-part:

1. `scripts/prefetch-reppo.sh` (or the cache layer) should record each
   pod's current-epoch eligibility — e.g. via `reppo query pod <id>`
   per pod, or a bulk endpoint if Reppo exposes one — and write a
   `valid_for_epoch: bool` field per pod into
   `.reppo-cache/pods-tradinggymai.json`.
2. `skills/reppo-trading-agent/SKILL.md` should require the agent to
   filter out pods with `valid_for_epoch: false` before selecting vote
   candidates.

Either part on its own is insufficient: the agent needs the data, and
without the prompt rule the agent will still consume all pods in the
cache.

## 2026-05-30 occurrence — epoch-91 depth probe confirmed boundary

1st run of 2026-05-30 queued 3 DISLIKE vote intents (pods 361 epoch 96,
299 + 297 epoch 91). pod 361 landed clean on-chain (tx 0x9e435761…);
pods 299 + 297 dry-run reverted `POD_NOT_VALID_FOR_EPOCH` as expected.
The trading-agent flagged this as a deliberate probe — it had exhausted
all unvoted epoch-95+ rubric-NO targets after the 4-day vote-pod sweep
(332/390/391/331/344/345/363/364/365/300/362/366/361 all landed), and
reached down into epoch 91 to confirm whether ISS-005's depth horizon
is strict. It is: epoch 91 (3 epochs behind) reverts. The agent's
organic strategy of cycling through pods has now visited every available
recent-epoch pod once; future runs without a prefetch-side filter will
either re-vote already-touched pods (compounding ISS-005's duplicate-vote
risk) or re-probe epoch 91 and burn more dry-runs. Durable fix priority
should rise accordingly.
