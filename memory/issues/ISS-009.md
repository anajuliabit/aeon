## Corrected diagnosis (2026-05-28, 2nd run of the day)

Root cause traced for the first time. The chain-runner's "Capture
skill output" step in `.github/workflows/aeon.yml:479-493` extracts
the Claude CLI's `.result` field (the final assistant text response)
to `/tmp/skill-result.txt` and then `cp`s that file over
`.outputs/${SKILL}.md`, **silently overwriting any Write-tool
output the orchestrator may have produced**.

So today's 1st-run flow was:
1. Orchestrator skill runs; LLM emits its `## Summary` prose and
   also calls the Write tool to write a fenced ```reppo-plan```
   block to `.outputs/reppo-orchestrator.md`.
2. Capture step in `aeon.yml:479-493` extracts the LLM's `.result`
   (the prose-only assistant text response with no fenced block).
3. Capture step `cp`s `/tmp/skill-result.txt` over
   `.outputs/reppo-orchestrator.md`, clobbering the Write-tool
   output.
4. Trading-agent reads the clobbered file, finds no fenced block,
   SKIPs.

**PR #24 and PR #27 targeted the wrong layer.** The orchestrator
was never the truncator. The contract that has to hold is "fenced
block in the LLM's final assistant text response," not "fenced
block on disk via Write tool" — because the disk file is always
overwritten with the assistant text at the end.

**Verification (this run, 2nd-run manual):** orchestrator emitted
the fenced block in its final assistant text response (not via
Write tool). Capture step propagated it unchanged. Trading-agent's
Step 1 gate cleared. 2 DISLIKE vote intents queued for pods
372/373. Postprocess landed both on-chain (tx
0xb2400bbc17362fcd865489b51bb817464850d0b50b8a6f839dd1306ed9b53d11
and tx 0x8f3130b6c5d0bbd4125e5a8b99df09daa2dd3f1662d2b404ab3f9a2151ccb281).

**Remaining work to close ISS-009:**

1. **Update `skills/reppo-orchestrator/SKILL.md`** to mandate the
   fenced block appears in the final assistant text response, and
   to explicitly say "do NOT use the Write tool for this — the
   capture step in `aeon.yml:479-493` will overwrite it."
2. **Still apply the chain-runner.yml fail-fast `continue` → `break`
   fix.** Even with the corrected emission path, the validator
   should hard-abort the chain on a missing block instead of
   silently letting the next step dispatch (3rd item in this issue's
   prior escalation list).
3. **Consider refactoring the capture step** to merge Write-tool
   output with assistant text rather than overwrite — but this is a
   chain-wide change and out of scope for this issue.

Status bumped from `open` to `investigating` (root cause traced,
fix path scoped). Will move to `fixing` once the orchestrator skill
update lands.

## Recurrence (2026-05-28, 1st run of the day)

4th occurrence overall (1st + 2nd 2026-05-26, 3rd 2026-05-27, 4th
today). 1st **post orchestrator-side Write-tool workaround** —
the orchestrator skill was modified to write the fenced block
directly via the Write tool instead of relying on the chain-runner
to capture stdout, and its `## Summary` for today's run explicitly
asserts "fenced block is the very first content in
`.outputs/reppo-orchestrator.md`, written directly via Write —
verified post-write that the prior run's file contained no fenced
block despite claiming otherwise (the 3rd ISS-009 occurrence)".

Yet the chain-passed `.outputs/reppo-orchestrator.md` still
contains only the `## Summary` section, no fenced ```reppo-plan```
block anywhere. The trading-agent's Step 1 gate failed and it
SKIPped — 0 intents queued, no `## Execution Results`, 0 on-chain.

This rules out the orchestrator's emission path as the truncation
point: the orchestrator wrote the file via the Write tool (an
atomic file write, not stdout-capture), claimed verification
post-write, and the file as seen downstream still lacks the
block. So the loss happens **after** orchestrator's Write
call, somewhere in:

- the chain-runner's `.outputs/` propagation between steps (most
  likely candidate — the chain-runner pulls fresh `.outputs/`
  state between steps, but if the orchestrator's auto-commit
  rewrites the file via `git add`/`git commit` from a stale
  pre-Write working copy, or if the chain-runner re-checks-out
  from a stale ref, the post-Write content gets clobbered);
- a post-step hook (notify-jsonrender, action-converter, or
  similar) rewriting `.outputs/reppo-orchestrator.md`;
- the chain-runner reading from a sibling/cached path rather
  than the working-tree file.

**Counts: 4 of 23 chain runs since 2026-05-26 morning** —
1st + 2nd occurrences halved on-chain throughput on 05-26,
3rd zero'd it on 05-27, 4th zero'd it again today. PR #24
prompt-tightening, PR #27 workflow-level grep guard, and the
orchestrator Write-tool workaround have all been bypassed —
prompt-level and orchestrator-side fixes are exhausted; the
remaining root cause sits in the chain-runner.

**Re-escalation (post Write-tool workaround):**

1. **The chain-runner.yml fail-fast `continue` → `break` change
   remains the unblocking step.** Even with the validator
   running and detecting the missing block, the `continue` in
   the `if [ "$ON_ERROR" = "fail-fast" ]` branch of the steps
   loop (~line 308 of `.github/workflows/chain-runner.yml`)
   silently skips to the next `for i` iteration rather than
   aborting — so the validator's failure doesn't stop the
   trading-agent dispatch. Switch to `break` (or `exit 1`).
2. **Audit `.outputs/` propagation between steps.** Confirm
   the file the trading-agent step reads is the post-orchestrator
   working-tree copy, not a re-checkout of a stale ref or a
   pre-Write captured stdout. The orchestrator now writes the
   file directly, but if the chain-runner's auto-commit step
   stages a stale pre-Write copy, downstream steps will fetch
   the stale version.
3. **Belt-and-suspenders at the trading-agent level.** The
   agent already SKIPs cleanly. Add a heartbeat-watch flag so
   the next reflect cycle escalates the day's reppo throughput
   loss without waiting for reppo-digest.

  no `## Execution Results`;
- chain produced 0 on-chain;
- the chain workflow nonetheless dispatched trading-agent — so the
  validator either did not run, ran after dispatch, or the chain's
  `fail-fast` branch (line 306–309 of `chain-runner.yml`) uses
  `continue` which skips to the next `for i` iteration of the *steps*
  loop rather than aborting the chain, letting the broken hand-off
  propagate downstream.

That's 3 of 22 chain runs since 2026-05-26 morning. The 2 previous
occurrences halved on-chain throughput; this one zero'd it out.

**Escalation path (post-PR-#27 re-escalation)**:

1. **Fix the chain-runner abort semantics.** The `continue` in the
   `if [ "$ON_ERROR" = "fail-fast" ]` branch of the steps loop
   (~line 308 of `.github/workflows/chain-runner.yml`) does not
   abort — bash `continue` skips to the next iteration of the `for`
   loop the statement is lexically inside, which here is the steps
   loop. The downstream step dispatches anyway. Switch to `break`,
   or refactor so that a validator-failed step prevents subsequent
   steps from dispatching.
2. **Verify validator ordering.** `validate_skill_output()` is
   called *after* `wait_for_runs` (line 322–330). Confirm the
   .outputs file the validator reads is the post-orchestrator
   committed version, not a stale pre-orchestrator copy. If
   chain-runner pulls latest only inside the `if [ ${#RUN_IDS[@]} -gt 0 ]`
   block (line 312–313) that should be fine, but worth double-checking
   the chain definition uses `parallel` / `sequential` correctly so
   trading-agent doesn't dispatch in the same step as orchestrator.
3. **Belt-and-suspenders at the trading-agent level**: the agent
   already SKIPs cleanly when the block is missing. Consider having
   it open a fresh issue / append a heartbeat-watch flag so the next
   reflect cycle escalates the day's reppo throughput loss without
   waiting for reppo-digest.

## What happened

Today's 18th `reppo-swarm` chain run dispatched. `reppo-orchestrator` ran
and recorded a clean `1 RUN / 0 SKIP` in `memory/logs/2026-05-26.md` and
in its own `## Summary`. But the chain-passed body in
`.outputs/reppo-orchestrator.md` contains *only* the Summary section —
no fenced ```reppo-plan``` block.

`reppo-trading-agent` runs next, gates on the fenced block per its
SKILL.md Step 1, and SKIPs:

```
Skipped: no orchestrator plan available
```

No intents queued, no `## Execution Results` section, no on-chain
activity. The 17th run earlier today (5th of today's reppo-swarm runs)
had the block parse correctly — the mint
`0x77f1386f…e48a61f` landed (first on-chain ever) on that run. So the
orchestrator's output format isn't permanently broken; this is an
intermittent drop on chain hand-off.

## Impact

- 1 entire reppo-swarm run lost (0 minted intents queued, 0 votes
  considered). For datanet 9 the practical impact is small because the
  mint cap is already saturated (1 minted today), but on a run where
  votes could land (post-ISS-006), losing the body would cost the
  vote slot.
- Pattern repeats undetected unless reppo-digest specifically calls it
  out — orchestrator + trading-agent both look "successful" in
  cron-state since each individual skill exits clean.

## Fix shipped (this PR)

Diagnosis: the orchestrator's emitted output itself dropped the
fenced block — verified by reading commit `ac7b950`
(orchestrator's auto-commit for run 18). Body was only a
`## Summary` section starting with `Executed reppo-orchestrator.
Read all rubrics + cache + ledger; produced the RUN/SKIP
plan…` — casual-report tone, no fenced block, no
`reppo-trading-agent: RUN` directive. This rules out
investigation path 2 (chain-runner truncation); the chain-runner
faithfully passed what the orchestrator emitted.

Fix on the orchestrator side (skill / prompt level):

1. **Moved the output contract to the top of `skills/reppo-orchestrator/SKILL.md`** as a non-negotiable section, before
   `## Inputs` and `## Steps`. The LLM sees it before it absorbs
   the rest of the skill.
2. **Tightened the language** — "MUST emit, every run, no exceptions, no shortcuts" replacing "ensure it ends with".
3. **Explicit anti-patterns** — "do NOT skip this block because
   the prior run already logged the plan / do NOT emit only a
   Summary section / do NOT replace the fence with a sub-section
   heading".
4. **Cited the regression directly** so the LLM has a concrete
   negative example to anchor against.
5. **Step 3 simplified** to reference the top-section contract
   rather than duplicating the format.

Escalation path if it recurs: workflow-level guard — grep
`.outputs/reppo-orchestrator.md` for the fenced block after the
orchestrator step, fail the step if absent so the chain doesn't
propagate a broken hand-off (instead of silently producing a 0-on-
chain run as today).

- Intermittent: 1 of 5 runs today (20%). Not at >50% threshold.
- The orchestrator + trading-agent + digest each still exit clean

- The first on-chain mint already landed earlier today (17th run),
