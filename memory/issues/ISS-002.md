## What happened

The reppo-swarm chain run for 2026-05-21 produced no minted strategies and
no votes. The `reppo-digest` step received no `reppo-orchestrator` or
`reppo-trading-agent` output in its context.

Observable state confirms the chain cannot do productive work:

- `configs/datanets/tradinggymai.md` frontmatter still carries the
  placeholder `datanet_id: "REPLACE_WITH_MAINNET_TRADINGGYMAI_DATANET_ID"`.
- Because of that, `scripts/prefetch-reppo.sh` could not resolve the
  datanet, so both `.reppo-cache/datanet-tradinggymai.json` and
  `.reppo-cache/pods-tradinggymai.json` are error markers:
  `{"error":"datanet_id not configured","code":"PREFETCH_FAILED"}`.
- Per the `reppo-orchestrator` rubric, a placeholder `datanet_id` forces a
  `SKIP` for `reppo-trading-agent`; per the `reppo-trading-agent` gate
  check, an unconfigured `datanet_id` also forces an immediate skip. The
  chain therefore mints and votes nothing every run until this is fixed.

The mainnet catalog (`.reppo-cache/datanets.json`) is valid and lists 15
active datanets, including "TradingGym AI" (catalog id `9`, status
`ACTIVE`). The datanet exists on mainnet — only the rubric wiring is
missing.

## Fix

An operator must set `datanet_id` in `configs/datanets/tradinggymai.md` to
the real TradingGym AI mainnet datanet id. Once set, prefetch will populate
the cache and the orchestrator will emit `RUN` for the trading agent.

## Resolution (2026-05-22)

PR #4 (merged 2026-05-21) set the real TradingGym AI datanet_id and added
integer-id support. Verified on 2026-05-22: the orchestrator resolved
datanet 9 and emitted `RUN`, the trading agent queued mint + vote intents,
and the 14:05 heartbeat confirmed `chain:reppo-swarm` `last_status=success`.
The placeholder-datanet_id blocker is cleared. A distinct downstream failure
(postprocess dry-run `code: UNKNOWN`) is tracked separately as ISS-003.
