## What happened

The 2026-05-25 `reppo-swarm` run queued 3 vote intents for datanet 9
(pods 300, 363, 364, all DISLIKE). `scripts/postprocess-reppo.sh` ran
in `dry_run_only=true` mode. Votes 300 and 364 reverted with
`INSUFFICIENT_VOTING_POWER` (ISS-006), but vote-363 failed differently:

```
output: {"error":{"code":"INTERNAL_ERROR",
                  "message":"RPC Request failed.\n\nURL: https://mainnet.base.org\n
                            Request body: {\"method\":\"eth_call\",\"params\":[
                            {\"data\":\"0xbcc3f3bd000000000000000000000000b4ec41c93cf2f573f82d8f023b01637eb5db4c64\",
                             \"to\":\"0x0EFBE19Cb7B07D934D01990a8989E9CaA98b9009\"},
                            \"late...
```

This is the second time the digest has logged this exact pattern —
the 2026-05-24 run also had one vote (vote-345) hit a transient
`mainnet.base.org` RPC error mid-batch. Both runs had the rest of the
batch return normal (deterministic) dry-run errors, so this is not a
broad network/sandbox outage — it's the upstream Base public RPC
flaking on a single request.

## Impact

- One queued intent per affected run silently drops out of the
  on-chain pathway with no usable dry-run signal. Once ISS-004 and
  ISS-006 are resolved and real writes start happening, the same
  transient could cause a confirmed vote_cap intent to be lost rather
  than retried.
- Today the underlying votes would have reverted anyway (ISS-006), so
  no real intent was lost — but this issue must be fixed before real
  writes begin or the chain will drop traffic on Base RPC blips.

## Fix

Add retry-with-backoff to `scripts/postprocess-reppo.sh` around the
Reppo CLI dry-run + write calls. Suggested approach:

- Detect "RPC Request failed" / `INTERNAL_ERROR` shape distinct from
  deterministic CLI error codes (`PUBLISHER_LACKS_SUBNET_ACCESS`,
  `INSUFFICIENT_VOTING_POWER`, `POD_NOT_VALID_FOR_EPOCH`, etc.) — only
  retry the transient class.
- Retry up to 3x with exponential backoff (e.g. 2s / 5s / 10s).
- Optionally switch the Base RPC endpoint to a more reliable provider
  (Alchemy / QuickNode / Infura) by setting whichever env var the
  Reppo CLI exposes for RPC override — public `mainnet.base.org` has a
  known history of intermittent failures.
- If retries are exhausted, surface the failure to the digest just as
  it does today.
