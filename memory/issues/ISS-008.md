## What happened

Today's 16th `reppo-swarm` run on datanet 9 was the first run where the
publisher actually had subnet access (the ISS-004 auto-grant landed on-chain
this run, tx `0x1d6581a9b2a5a3c2b8080b1f1cf890e1abb280f1ec3d9a87fc79e571ba7a90dd`).
Immediately after the grant, the mint dry-run for
`mint-c471e15adfb0a009.json` (BTC perp 5m CVD-divergence long-squeeze
reversal short) reverted with a previously-unseen selector:

```
output: {"error":{"code":"UNKNOWN_REVERT_0x13be252b",
                  "message":"Simulation reverted",
                  "hint":"The contract function \"mintPodWithREPPO\" reverted with the following signature:\n0x13be252b"}}
```

This is not `PUBLISHER_LACKS_SUBNET_ACCESS` (ISS-004), not
`INSUFFICIENT_ALLOWANCE` (ISS-004 sub-blocker), not `POD_NOT_VALID_FOR_EPOCH`
(ISS-005), not `INSUFFICIENT_VOTING_POWER` (ISS-006), not the transient
`mainnet.base.org` RPC `INTERNAL_ERROR` (ISS-007), and not the CLI's
`code: UNKNOWN` wrapping bug (ISS-003, already fixed in PR #8). It is a
genuinely new failure mode that only became reachable because the
preceding ISS-004 sub-blockers cleared.

## Impact

- 0 strategies minted on-chain for this run (votes still gated by ISS-006).
- Until the selector is decoded and the underlying requirement is
  satisfied, every future mint intent for datanet 9 will revert at
  dry-run with the same code, regardless of strategy content. The chain
  is unblocked one layer (subnet access) but immediately blocked on the
  next.

## Fix

Two parallel paths:

1. **Decode the selector.** `0x13be252b` is the first 4 bytes of
   `keccak256(<errorName>(<argTypes>))`. Either:
   - Look up on Basescan: open the SubnetManager
     (`0x2629A8083065938B533b117704935D727270eE7A`) source / ABI and
     search the custom errors for any selector matching `0x13be252b`.
   - Cross-check against 4byte.directory.
   - Inspect Reppo CLI source for known custom errors in
     `mintPodWithREPPO`'s revert path that the CLI's error-code map
     hasn't been wired up to yet.
2. **Once decoded**, treat the underlying requirement the same way
   ISS-004 was treated — either an operator action (if a balance /
   approval / configuration prereq), or extend the auto-helper
   (if a programmatically-fixable allowance/state issue).

Candidate guesses worth checking first (all plausible given the
mintPodWithREPPO path on a freshly-granted publisher with the prior
approve tx of 50 REPPO at minimum):

- **`InsufficientREPPOBalance`** — publisher may have approved the
  SubnetManager but not actually hold the mint fee balance.
- **`MintCapExceeded`** / **`PodAlreadyExists`** — datanet 9's rubric
  has `mint_cap: 1`; if the publisher already has 1 minted pod on this
  datanet on-chain (independent of this chain's ledger), additional
  mints would revert.
- **`InvalidStrategyHash`** / **`HashCollision`** — contract-side
  validation we haven't seen exercised before.
- **`SubnetPaused`** / **`MintingDisabled`** — operational toggle.
