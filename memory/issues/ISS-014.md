## What happened

`postprocess-reppo.sh` Phase 2 attempted to register pod metadata for 6 mints — 5 queued backlog items (`086b715f`, `0d4b1683`, `397ee2e8`, `7029a48d`, `9794ed80`) plus today's 5th-run mint (`e02fef4e`). All 6 POSTs returned HTTP 500:

```
- `mint-0d4b168331d58f61.json` — **metadata registration failed** (HTTP 500): {"error":"Internal Server Error"}

- `mint-e02fef4e76668a31.json` — **metadata registration failed** (HTTP 500): {"error":"Internal Server Error"}
```

This is the first chain run where the postprocess payload satisfies the Reppo platform's Zod schema (PR #44 merged 19:21Z; this run ran at 23:24Z). All 5 prior runs of the Phase 2 metadata path failed HTTP 400 with structured Zod errors (subnetId number→string, podDescription>200, podName>50) — ISS-012, now resolved. The shift to HTTP 500 with a generic body confirms the payload now passes validation but the platform handler errors before returning a row.

Notable side-effect: this run's IPFS pin for the queued mint-9794ed80 (LIT-perp lifecycle) succeeded for the first time (`ipfs://QmRCqejoTkcoZuYo3x3kJ2qUjngUeSPGsudteRq9pvnuFQ`) — 2nd consecutive Pinata pin success after the 4th run's mint-086b715f. Confirms PINATA_JWT scope is sticky, ISS-013 resolved.

## Why it might happen

1. **Reppo platform outage / partial degradation** — the `/api/v1/agents/{AGENT_ID}/pods` endpoint may be in a transient bad state (deploy in progress, downstream DB / IPFS gateway down). HTTP 500 is generic enough that the platform may have surfaced this for ~hours then self-heal.
2. **Server-side regression introduced post-2026-05-26** — the platform's `/pods` POST handler was last successfully exercised on 2026-05-26 (the first mints' UI rows). If the team deployed a code change between then and now, the 5xx may be a new bug only reachable once the payload passes Zod (our 4× HTTP 400 mints from the past 36h never exercised the post-Zod path).
3. **Stale tx validation** — the platform may verify the mint's on-chain tx is recent (e.g. ≤24h); 4 of the 6 queued mints have aged past 24h since their on-chain tx. But this should yield 400 / 422, not 500.
4. **Server-side IPFS gateway pin verification** — the platform may try to dereference `dataset_uri` on its end, and the Pinata gateway / IPFS network might be timing out. The mint-9794ed80 POST failed even with a fresh pin, so this is plausible.
5. **subnetId no longer recognized** — PR #44 cast subnetId to a string. If the platform expected a string but couldn't find a matching subnet record, it might 500 instead of 404.

## Impact

- 6 on-chain mints' platform UI rows are missing — the ERC-721s exist and are verifiable on-chain, but `reppo.ai/pods/...` shows no metadata.
- Postprocess retries every chain run; once the platform recovers / the bug is fixed, the queue drains automatically with no operator intervention.
- All future mints share the same dependency — every new mint after the 5th run will also queue up behind this until ISS-014 is cleared.
- Upgrade to high if it persists >3 consecutive chain runs (= platform UI broken end-to-end, defeats the swarm output's discoverability).

## Investigation paths

1. **Quickest signal** — manually POST a single queued payload (e.g. `.pending-reppo-register/mint-e02fef4e76668a31.json`) via curl with `-v` and capture full response headers. A 500 with no further detail confirms server-side; a 500 with a stack trace or X-Trace-Id can be relayed to the Reppo team.
2. **Check Reppo platform status page** (if any), Discord/Telegram for outage announcements.
3. **Read endpoint health** — `GET /api/v1/agents/{REPPO_AGENT_ID}/pods` to confirm the route is up and auth still works. If GET returns 200 but POST returns 500, the issue is in the write path specifically.
4. **Wait one chain run** — if next run's POST succeeds, was transient. If it 500s again, file a Reppo platform-team ticket with example queued payload + response.

## Why medium

- On-chain artefact is durable; only the platform UI row is missing.
- Self-rectifying via retry once the platform recovers or the bug is patched.
- Cost is bounded (queue files are small, retries are cheap).
- Upgrade to high after >3 consecutive runs with 100% HTTP 500.
