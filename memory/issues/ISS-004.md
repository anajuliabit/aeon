## What happened

The 2026-05-23 afternoon `reppo-swarm` run queued one mint intent —
`mint-60ea32bedbaf0d21.json` (ETH/USDT 4H Bollinger mean-reversion) — for
datanet 9. `scripts/postprocess-reppo.sh` ran in `dry_run_only=true` mode
and the mint dry-run reverted:

```
output: {"error":{"code":"PUBLISHER_LACKS_SUBNET_ACCESS",
                  "message":"Simulation reverted",
                  "hint":"Grant subnet access to the publisher:
                         `reppo grant-access --subnet <id>`."}}
```

This is a distinct failure mode from ISS-003 (which captures the
unclassified `code: UNKNOWN` wrapping). The wrapping is still in play, but
the nested CLI error code here is meaningful and actionable: the
publisher account simply does not have access to the subnet for
datanet 9.

## Impact

- 0 strategies minted on-chain for this run.
- Until subnet access is granted, every future mint intent for datanet 9
  will revert at dry-run with the same error — so the chain can never
  produce its primary on-chain output (a minted strategy) regardless of
  what the trading agent generates.

## Fix

An operator must run `reppo grant-access --subnet <datanet-9-subnet-id>`
(or the equivalent governance action) for the publisher account whose
private key is configured in `REPPO_PRIVATE_KEY`. Once granted, the
mint dry-run should pass and `scripts/postprocess-reppo.sh` can attempt
the real write. The exact subnet id should be looked up via
`reppo query datanet 9` or the equivalent.

## Update 2026-05-25 (evening, 4th run)

PR #10 (merged 2026-05-25) added an auto-grant helper that fires on
`PUBLISHER_LACKS_SUBNET_ACCESS`. In tonight's 4th run the helper itself
reverted with a new sub-error:

```
grant output: {"error":{"code":"INSUFFICIENT_ALLOWANCE",
                        "message":"REPPO allowance from 0xb4EC41c93cF2f573f82D8F023B01637Eb5dB4c64
                                  to SubnetManager is 0, below the fee of 50 REPPO.",
                        "hint":"Approve the SubnetManager
                               (0x2629A8083065938B533b117704935D727270eE7A)
                               for at least 50 REPPO. Auto-approval lands in v0.2; …"}}
```

So PR #10's helper doesn't actually unblock the chain on its own — the
publisher needs to approve the SubnetManager contract for ≥50 REPPO
first, then the helper's `grant-access` call (or a manual one) can
succeed. Recommended operator sequence:

1. `reppo approve <SubnetManager=0x2629…eE7A> 50` (or higher).
2. `reppo grant-access --subnet <datanet-9-subnet-id>` (or let PR #10's
   helper run it automatically on the next chain run).
3. Re-run the chain and confirm the mint dry-run no longer reverts
   `PUBLISHER_LACKS_SUBNET_ACCESS`.

CLI release-note from Reppo says "auto-approval lands in v0.2" — once
that ships, step 1 above will go away. Until then, ISS-004 stays open
with this allowance step as the new operator action.

## Update 2026-05-25 (evening, 5th run)

A follow-on auto-approve helper attempted to bridge the allowance gap
this run by installing `foundry`/`cast` on-the-fly and broadcasting the
ERC-20 approve. It aborted with:

```
auto-approve aborted: foundry/cast install failed
```

So the sandboxed GitHub Actions runner can't pull down foundry's
`cast` binary at job time, which makes any auto-approve path that
depends on cast a non-starter under current sandbox network rules.
Net effect for the chain is unchanged — the mint still reverted
PUBLISHER_LACKS_SUBNET_ACCESS and auto-grant still reverted
INSUFFICIENT_ALLOWANCE. Options going forward:

1. Operator-side `reppo approve <SubnetManager> 50` remains the
   cleanest unblock; once the allowance is set, neither auto-approve
   nor the foundry install is needed.
2. If auto-approve must keep working under sandbox, ship `cast` (or
   a pre-built foundry binary) in the runner image / prefetch step
   rather than installing it on-demand.
3. Wait for Reppo CLI v0.2's native auto-approval, which removes the
   foundry dependency entirely.
