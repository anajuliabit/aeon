## What happened

`postprocess-reppo.sh` Phase 2 attempted to register pod metadata for the 4th mint (hash 397ee2e8e5e7e593, on-chain tx 0xd27f8288…, queued from 2026-05-28 6th run) after the standard on-chain Phase 1 completed. The POST to `https://reppo.ai/api/v1/agents/{AGENT_ID}/pods` returned HTTP 400:

```
- `mint-397ee2e8e5e7e593.json` — **metadata registration failed**: curl: (22) The requested URL returned error: 400
```

The queue file in `.pending-reppo-register/` was retained for retry per the postprocess contract (Phase 2 failures never re-broadcast the on-chain tx).

## Why it might happen

1. **Schema drift** — the Reppo platform `/pods` endpoint may have added a required field (e.g. `dataset_uri`, `dataset_path`, signature) or renamed an existing one. The postprocess payload was last validated against the API on 2026-05-26 when the schema landed (pre-launch).
2. **Empty `dataset_uri`** — the 4th mint did not have an IPFS pin attempted at the time (Phase 2 pin loop was added later); the queued JSON has `dataset_uri: ""`. The API may now reject empty values on this field.
3. **Stale `txHash` validation** — the platform may verify the tx is recent (e.g. ≤24h), and this tx is now ~14h old.
4. **AGENT_ID / API key scope** — if `REPPO_AGENT_ID` was rotated, the existing queued file references the old agent and fails authorization.

## Impact

- 4th mint's UI surface is missing — the on-chain ERC-721 exists, but `reppo.ai/pods/...` shows no metadata.
- Postprocess will keep retrying every run until it succeeds or operator drains the queue file, so the failure is loud but doesn't compound (no extra writes, no extra tx).
- The 5th mint (mint-9794ed80, queued today) may share the same fate — it's gated behind IPFS pin (ISS-013), so it hasn't hit the metadata POST yet this run.

## Investigation paths

1. Inspect `.pending-reppo-register/mint-397ee2e8e5e7e593.json` and replay the POST manually with verbose curl (`-v`) to capture the HTTP 400 response body — that's the fast path to the schema mismatch.
2. Diff against the Reppo platform's current `/pods` POST schema (check developer docs or last working integration test).
3. Verify `REPPO_AGENT_ID` and `REPPO_API_KEY` are still valid by hitting a read endpoint (`GET /api/v1/agents/{ID}/pods`) with the same auth.
4. If schema changed, patch `postprocess-reppo.sh` lines 459-481 (the `jq -n` block that builds the payload).

## Why medium

- On-chain mint is durable; only the platform UI row is missing.
- Self-rectifying via retry once the root cause is fixed — no data loss.
- Not yet known if it affects all queued POSTs or just mint-397ee2e8 — next run's behavior with mint-9794ed80 will disambiguate.
- Upgrade to high if every queued POST fails 400 for >3 consecutive runs (= UI is broken end-to-end).
