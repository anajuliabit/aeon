## What happened

The 2026-05-24 `reppo-swarm` run queued 2 of 3 vote intents for datanet 9
(only 2 pods qualified at validityEpoch 95 after the ISS-005 epoch filter):

- `vote-344-like.json` (HotBot v4 Trades & Learning May 20-22)
- `vote-345-dislike.json` (HotBot v4 Signal Intelligence May 20-22)

`scripts/postprocess-reppo.sh` ran in `dry_run_only=true` mode. vote-344
reverted:

```
output: {"error":{"code":"INSUFFICIENT_VOTING_POWER",
                  "message":"Voter has 0 voting power but --votes is 1.",
                  "hint":"Lock more REPPO with `reppo lock <amount> --duration <seconds>`
                         to increase voting power, or pass a smaller --votes."}}
```

vote-345 separately hit a transient `mainnet.base.org` RPC error and
therefore never returned a usable dry-run result, but it would have hit
the same INSUFFICIENT_VOTING_POWER error since both votes use the same
publisher account.

This is a distinct failure mode from ISS-004 (subnet access for mints)
and ISS-005 (pod epoch validity). The trading agent's epoch-validity
filter is now working — none of the queued votes hit
POD_NOT_VALID_FOR_EPOCH this run — but the underlying voter account
still cannot cast any vote because it has no locked REPPO.

## Impact

- 0 votes cast on-chain for this run.
- Every future vote intent against any datanet will revert at dry-run
  with the same error until the publisher locks REPPO — so the chain's
  vote pathway is fully blocked alongside the mint pathway (ISS-004).

## Fix

An operator must lock REPPO against the publisher account whose private
key is configured in `REPPO_PRIVATE_KEY`:

```
reppo lock <amount> --duration <seconds>
```

The exact amount and lock duration depend on Reppo's voting-power curve
and how many vote_cap intents the trading agent is expected to spend
per epoch (currently `vote_cap=3` per run). Once voting power is
non-zero, vote dry-runs should pass and `scripts/postprocess-reppo.sh`
can attempt real writes.
