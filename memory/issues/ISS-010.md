## What happened

This Claude run (2026-05-28 morning) was dispatched with prompt
`Read and execute the skill defined in skills/reppo-swarm/SKILL.md`.
That file doesn't exist — `reppo-swarm` is a chain, defined in
`aeon.yml:299-308`, not a skill. Same prompt fired on 2026-05-27 (see
`.outputs/reppo-swarm.md` left over from that run) and stopped at the
same spot.

## Why it happens

`.github/workflows/messages.yml:109-138` builds the SKILL_ORDER list
by iterating every 2-space-indented YAML key in `aeon.yml`, with no
section boundary. The chains parser later in the file (lines 242-247)
does scope correctly to the `chains:` block; the skills parser does
not scope to `skills:`. Net effect: every key under `chains:`
(currently just `reppo-swarm:` at line 300) is double-classified as a
skill AND a chain.

The chain dispatch (line 313 via `gh workflow run chain-runner.yml -f chain="$CH"`)
runs the real chain correctly. The phantom skill dispatch (via
`gh workflow run aeon.yml -f skill="$SKILL"`) sends Claude into
`skills/reppo-swarm/SKILL.md` which doesn't exist.

## Impact

- Wasted Claude run every morning at ~07:00 UTC (one extra workflow
  job, ~$ in tokens, ~minutes of runner time).
- Cron-state pollution: a phantom `reppo-swarm` key with always-clean
  success_rate, sitting alongside the real `chain:reppo-swarm` key.
  Confuses heartbeat / skill-health audits that key off
  cron-state.
- Silent: workflow exits 0, no notification fires, no logs flag it.
  Discovered only when investigating the stray `.outputs/reppo-swarm.md`.

## Fix

PR scopes the SKILL_ORDER parser to the `skills:` block, mirroring
the existing scoping pattern used by the chains parser at
lines 242-247:

```bash
IN_SKILLS=false
while IFS= read -r line; do
  if [[ "$line" =~ ^skills: ]]; then IN_SKILLS=true; continue; fi
  $IN_SKILLS && [[ "$line" =~ ^[a-z] ]] && IN_SKILLS=false
  $IN_SKILLS || continue
  # ...existing match logic...
done < aeon.yml
```

After this lands, the `reppo-swarm` cron-state key will stop
accruing new dispatches. The stale entry is left in place — no
cleanup needed, it just stops updating.

## Why medium

- No on-chain impact. The real `chain:reppo-swarm` continues to
  fire correctly via chain-runner.yml — yesterday's full chain ran
  clean (`chain:reppo-swarm` last_success 2026-05-26T21:55:51Z per
  yesterday's heartbeat).
- Cost is bounded (one extra Claude run/day, exits fast).
- Not user-visible. But it's a real waste and it pollutes the
  health surface, so worth fixing now while it's the only
  chain — the cost scales linearly with the number of chains
  defined.
