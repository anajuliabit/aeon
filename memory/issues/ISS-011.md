## What happened

`postprocess-reppo.sh` ran the 4-intent batch (1 mint + 3 votes) for the 2026-05-29 1st run. Mint and two votes (pods 332, 390) succeeded on-chain. The 3rd vote (pod 391) reverted:

```
- `vote-391-dislike.json` — **write failed** (code: REVERT)
  - output: {"error":{"code":"REVERT","message":"Vote tx failed to submit","hint":"Nonce provided for the transaction (119) is lower than the current nonce of the account."}}
```

This is the first nonce-related failure since the chain went live 2026-05-26 — distinct from ISS-007's transient Base RPC INTERNAL_ERROR pattern.

## Why it might happen

Three plausible causes:

1. **CLI per-process nonce caching** — `reppo vote-pod` reads `eth_getTransactionCount(latest)` once at start, increments locally between intents in the same batch, but doesn't reconcile if an out-of-band tx (auto-approve, auto-lock, external use of the publisher key) advanced the chain nonce mid-batch.
2. **Race with auto-recover helpers** — the postprocess auto-approve / auto-grant / auto-lock helpers (`reppo approve`, `reppo grant`, `reppo lock`) all broadcast txs as the publisher. If one of these fires between votes 390 and 391 without the CLI seeing the receipt, the local nonce counter falls behind.
3. **Replay protection from a prior partial broadcast** — vote-391 was previously broadcast and reverted (block reorg, pending mempool), and the CLI re-used the original nonce instead of fetching latest.

## Impact

- 1 vote intent lost per occurrence. Not destructive — the pod can be re-voted next run (would be a duplicate by ISS-005 standards but the vote itself is benign).
- Postprocess rm's the intent file on write-fail, so no auto-retry — fix has to come from the CLI or from postprocess-side retry logic.
- ISS-005 makes vote-write idempotency already broken, so a fresh tx on re-vote next run is the same risk profile already accepted.

## Investigation paths

1. Check whether `reppo vote-pod` accepts a `--nonce` flag or honors `REPPO_NONCE_STRATEGY=pending` (re-query before each write). If yes, set in `postprocess-reppo.sh`.
2. Inspect `.reppo-cache/result-vote-391-dislike.json` for the full error envelope — does it include the chain's current nonce so we can confirm the gap?
3. If reproducible, add a postprocess-side retry that re-runs the failed intent after a 5-10s delay (chain nonce should reconcile by then).
4. Coordinate with auto-recover helpers: serialize their broadcasts behind a lock so they can't race the main intent batch.

## Why medium

- Single intent lost per occurrence; pod is re-votable next cycle.
- Not yet reproducible — 1 occurrence across 24 chain runs.
- No silent failure: the REVERT is loud in `## Execution Results`.
- Upgrade to high if recurrence rate >20% over the next 5 runs.
