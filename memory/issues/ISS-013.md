## What happened

`postprocess-reppo.sh` Phase 2 attempted to pin the 5th mint's dataset to IPFS before posting metadata to the Reppo platform. The pin failed:

```
- `mint-9794ed8044e6e7ea.json` — **IPFS pin failed**: curl: (22) The requested URL returned error: 403 (queue file retained for retry)
```

This is the first IPFS pin attempt in the chain's history (the 4 prior mints predate the Phase 2 pin loop or had an empty dataset path). Because the pin failed, the metadata POST for this mint was skipped this run (the postprocess gates the POST on a successful pin or an explicitly absent dataset).

## Why it might happen

1. **`PINATA_JWT` missing** — the secret may not be wired into the chain-runner workflow env for the postprocess step. `postprocess-reppo.sh:554` handles the explicitly-unset case (`PINATA_JWT not set; POST proceeds without IPFS dataset link`) but a *present-but-invalid* JWT would surface as 403 instead.
2. **JWT expired or revoked** — Pinata JWTs can be issued with TTLs; if the integration was set up with a short-lived JWT, it's overdue for rotation.
3. **JWT scope** — the JWT may lack the `pinFileToIPFS` permission (Pinata scopes are per-route).
4. **Account quota** — Pinata's free tier caps pin count and storage. Hitting either returns 403.

## Impact

- 5th mint's dataset is not on IPFS — the platform metadata POST can't link a `dataset_uri` until the pin succeeds.
- Pin retries every run; once the JWT / quota is fixed, the queue drains automatically with no operator intervention beyond the secret rotation.
- All future mints share this dependency — if `PINATA_JWT` is structurally broken, every subsequent mint will hit the same 403.

## Investigation paths

1. Confirm `PINATA_JWT` is set in the chain-runner workflow env (`echo "${PINATA_JWT:+set}${PINATA_JWT:-unset}"` in a debug run, or check repo secrets).
2. If set, validate with a no-op call: `curl -H "Authorization: Bearer $PINATA_JWT" https://api.pinata.cloud/data/testAuthentication` — returns 200 if valid, 403 if not.
3. Check Pinata account dashboard for quota (pin count, total bytes pinned, monthly bandwidth).
4. If JWT is invalid, rotate via the Pinata console and update the GitHub Actions secret.
5. Consider switching to web3.storage / Filebase / a self-hosted IPFS node if Pinata's free-tier ceiling is too low for the projected mint cadence (one mint per chain run, ~5-15 KB per dataset for trade lifecycles, up to 2000 rows).

## Why medium

- On-chain mint is durable; only the dataset link in platform metadata is missing.
- Self-rectifying via retry once the secret is fixed.
- Cost is bounded (queue file is small, retry is fast).
- Upgrade to high if it blocks every mint for >3 consecutive runs — at that point the chain is producing on-chain artefacts with no associated dataset visible to consumers, which defeats the point of the swarm.
